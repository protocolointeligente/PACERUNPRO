### Pattern: Coach Lookup Desnecessário
**Problema Comum:** Fazer 1 query para buscar coach, depois usar `coach.id`
```typescript
// ❌ Antes: 2 queries
const coach = await prisma.coach.findUnique({ where: { userId } });
const data = await prisma.something.findMany({ where: { coachId: coach.id } });

// ✅ Depois: 1 query
const data = await prisma.something.findMany({ 
  where: { coach: { userId } } 
});
```

## 🔧 Técnicas de Refactoring Aplicadas

### Técnica 1: Direct Relationship Query
```typescript
// Antes
const coach = await prisma.coach.findUnique({ where: { userId } });
const items = await prisma.item.findMany({ where: { coachId: coach.id } });

// Depois
const items = await prisma.item.findMany({ 
  where: { coach: { userId } } 
});
```

### Técnica 2: Connect ao Criar
```typescript
// Antes
const coach = await prisma.coach.findUnique({ where: { userId } });
const item = await prisma.item.create({
  data: { coachId: coach.id, ...rest }
});

// Depois
const item = await prisma.item.create({
  data: { 
    coach: { connect: { userId } },
    ...rest 
  }
});
```

### Técnica 3: Batch Query para N+1
```typescript
// Antes (N+1)
for (const coach of coaches) {
  const sub = await prisma.subscription.findFirst({
    where: { userId: coach.user.id },
    orderBy: { startedAt: 'desc' },
    take: 1
  });
}

// Depois (1 query)
const subs = await prisma.subscription.findMany({
  where: { userId: { in: userIds } },
  distinct: ['userId'],
  orderBy: { startedAt: 'desc' }
});
```

## ⚠️ Considerações Importantes

1. **Prisma ORM:** Usa lazy loading por padrão. Relationships diretas requerem `select` ou `include` explícito.

2. **Connection Caching:** Browser pode cachear resultados. Validação importante em PUT/PATCH/DELETE.

3. **Batch Operations:** Para N > 50 items, considerar paginação ao invés de fetch tudo.

4. **Future:** Considerar caching em Redis para queries frequentes (admins, leads, etc).
